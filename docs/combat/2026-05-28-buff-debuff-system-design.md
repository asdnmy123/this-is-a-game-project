## 一、分类体系

所有效果分为四大类，每类的行为规则不同。

### 1.1 四大类总览

| 大类 | 子类 | 例子 | 核心规则 |
|------|------|------|---------|
| **属性类** | 面板属性修正 | ATK+30%、DEF+20%、SPD-20%、CRIT+15% | 影响 Step 0 面板；同类取最高，异类共存 |
| **状态类** | 特殊行为标记 | 嘲讽、护盾、隐身、免疫 | 独立于数值系统的行为标记 |
| **持续类** | 每轮生效 | DOT（灼烧/中毒）、HOT（回春/再生） | 轮开始/结束阶段触发；可叠加层数 |
| **控制类** | 限制行动 | 眩晕、沉默、冰冻、睡眠 | 禁止特定行动类型；控制互斥 |

### 1.2 效果标记体系

每个效果实例带以下标记：

| 标记 | 值 | 说明 |
|------|-----|------|
| `type` | `buff` / `debuff` | 正/负效果标记。净化的目标选择仅看此标记 |
| `category` | `stat` / `status` / `dot` / `hot` / `control` | 所属大类 |
| `cleanseable` | `true` / `false` | 是否可被净化/驱散 |
| `stack_rule` | `replace` / `refresh` / `layer` | 叠加时行为（见第三章） |

**Buff 和 Debuff 的判定原则：**
- 来自己方技能 → `buff`（正效果标记）
- 来自敌方技能 → `debuff`（负效果标记）
- 标记决定了"素问回春术是否净化这个效果"
- 特殊：自伤效果（如"自身损失 10% HP"）仍然是 debuff，但 source 是己方

## 二、生命周期

### 2.1 标准生命周期

```
施加 → 生效 → 每轮轮末计时 → 到期移除
```

**阶段详解：**

1. **施加** — 技能命中目标时创建效果实例
   - 如果目标已有同类效果 → 按叠加规则处理（第三章）
   - 新效果的剩余轮数 = 技能的持续轮数
   - 施加动作本身产生 30 仇恨（对全体敌人，仇恨系统 2.1 规定）

2. **生效** — 效果从创建的那一刻开始立即生效
   - 属性类效果：在下次 Step 0 计算面板时生效
   - 控制类效果：立即检查是否能阻止当前/后续行动
   - 持续类效果：在当前轮的下一个触发点生效

3. **计时** — 在**每轮结束阶段**统一减 1（回合流程 第四章）
   - `剩余轮数 -= 1`
   - 剩余轮数降至 0 时标记为"待移除"

4. **移除** — 在轮结束阶段的计时之后统一移除
   - 移除时不触发任何效果（除非有"消失时触发"的被动，预留）
   - 同时移除所有与该效果相关的数据

### 2.2 护盾作为特殊状态的生命周期

护盾不属于属性类也不属于持续类，它是**独立的伤害吸收层**。生命周期在技能系统 7.2 已定义，此处正式纳入本系统：

```
施加 → 护盾值附着在目标上 → 受伤优先扣除护盾 → 轮末计时 → 到期移除
```

护盾的特殊之处：
- 不参与 Step 0 面板属性计算
- 护盾值在伤害 Step 8（最终伤害）之后扣除
- 护盾值不是 HP，不受治疗影响

### 2.3 "持续到战斗结束"标记

- 剩余轮数设为 `-1`（特殊标记）
- 轮结束时不减
- 仅在战斗结束、目标死亡或特定驱散时移除

### 2.4 施放者阵亡后的处理

遵循回合流程文档的方案 A（推荐）：

**规则：施放者阵亡后，其施加的 buff/debuff 保留。** 理由：
- 避免玩家困惑"我刚上的 buff 怎么没了"
- 实现简单，不需要追踪每个效果的施放者存活状态
- 负面影响可控——没有哪个 buff 强到"没了就玩不了"

**例外：嘲讽。** 嘲讽效果与玄甲强绑定，玄甲阵亡后嘲讽立即消失（仇恨系统 7.4 已规定）。

## 三、叠加规则

### 3.1 叠加行为定义

每个效果定义一种叠加行为：

| 行为 | 规则 | 适用场景 |
|------|------|---------|
| `replace` | 新效果替换旧效果，旧值丢弃，新持续时间 | 护盾 |
| `refresh` | 新效果替换旧效果，取新值的 **持续时间**，效果值取**较高者** | 同类属性 buff/debuff |
| `layer` | 新效果叠加层数，层数影响效果强度（有上限） | DOT、HOT、部分可叠层被动 |

### 3.2 同类 Buff/Debuff 不叠加（refresh）

**核心规则（来自技能系统 10.1~10.2，正式化）：**

- **同 ID 的效果**触发 `refresh` 行为
- 效果值：取**较高者**（两个 DEF+20% 取 +20%，不是 +40%）
- 持续时间：重置为新效果的完整时长
- 被覆盖的效果不会"恢复"——旧效果在覆盖时完全丢失

**例子：**

```
玄甲铁壁嘲讽上 DEF+20%（持续 2 轮）
第 2 轮时再次使用铁壁嘲讽
→ DEF+20%（持续 2 轮，刷新）
→ 不会叠加为 DEF+40%
```

### 3.3 不同类效果共存

- 不同 ID 的效果同时生效，互不影响
- 例如：ATK+30%、DEF+20%、SPD-20% 可以同时存在于一个角色身上
- 在 Step 0 或 Step 7 中各自独立计算

**同一属性上的不同ID效果叠加：**
- 每个 buff 独立贡献其修正值，最终在 Step 0 代数相加
- 例如：铁壁嘲讽 DEF+20%（ID: BUFF_XJ_DEF_UP）与装备 DEF+10%（ID: BUFF_EQ_DEF）同时存在时
  - 并非"取最高 +20%"，而是各自贡献：最终 DEF% 修正 = (+20%) + (+10%) = +30%
- 不同 ID 意味着不同来源，没有理由互相覆盖——只有同 ID（同一技能再次施加）才触发 refresh 覆盖

### 3.4 Buff + Debuff 对冲计算

**当同一属性上同时存在 buff 和 debuff 时（来自技能系统 10.2，正式化）：**

```
最终修正百分比 = sum(所有 buff 的该属性修正) + sum(所有 debuff 的该属性修正)
```

- buff 为正值，debuff 为负值
- 两者代数相加（正负抵消）
- 不存在"buff 和 debuff 谁优先级高"——直接求和

**例子：**

```
惊鸿身上有：
  鹰眼（自 buff）：ATK +30%
  敌人 debuff：ATK -10%

Step 0 计算 final_ATK 时：
  ATK_percent_bonus = (+30%) + (-10%) = +20%
  final_ATK = base_ATK × (100 + 20) / 100
```

**如果 sum 为负：** 属性降低（如最终 ATK_percent_bonus = -15%，则 `final_ATK = base_ATK × 85%`）

### 3.5 护盾叠加（replace）

来自技能系统 7.1 规则 2：
- 新护盾替换旧护盾
- 旧护盾剩余值丢弃
- 持续轮数重置为新护盾的时长
- 不存在"叠加护盾"

### 3.6 DOT/HOT 叠加（layer）

详见第六章。

## 四、Layer 0 面板修正（与伤害公式 Step 0 对接）

### 4.1 哪些效果影响 Step 0

伤害公式 Step 0 提供了面板属性的修正框架：

```
final_ATK = base_ATK × (100 + ATK_percent_bonus) / 100 + ATK_flat_bonus
final_DEF = base_DEF × (100 + DEF_percent_bonus) / 100 + DEF_flat_bonus
最终 SP D、CRIT、CRIT_DMG 同理
```

**以下 Buff/Debuff 在 Step 0 生效：**

| 属性 | buff 示例 | debuff 示例 | 影响 |
|------|----------|------------|------|
| ATK | 鹰眼 ATK+30% | 虚弱 ATK-X% | 计算 final_ATK |
| DEF | 铁壁 DEF+20% | 破甲 DEF-X% | 计算 final_DEF |
| SPD | 疾风 SPD+X% | 缚灵咒 SPD-20% | 影响出手顺序排序 |
| CRIT | 鹰眼 CRIT+20% | 迷茫 CRIT-X% | 影响暴击判定 |
| CRIT_DMG | 锐目 CRIT_DMG+X% | — | 影响暴击倍率 |

**Step 0 修正的聚合计算规则：**

```
每个属性的最终 bonus 值 = 
  装备提供的 bonus（百分比） + 
  所有 active buff 提供的 bonus（百分比） - 
  所有 active debuff 提供的 penalty（百分比）

然后代入公式：
  final_ATK = base_ATK × (100 + ATK_percent_bonus) / 100
```

**实现注意：** 百分比修正和多段计算有可能产生中间小数，建议统一 `Math.floor`：

```javascript
final_ATK = Math.floor(base_ATK * (100 + atkPercentBonus) / 100);
```

### 4.2 Step 0 不包含哪些

- **DOT/HOT 伤害**（不在面板属性中，独立计算）
- **嘲讽/护盾/控制**（不涉及属性计算）
- **"造成的伤害 +X%"**（在 Step 7 计算）
- **"受到的伤害 +X%"**（在 Step 7 计算）

### 4.3 多项修正的聚合规则

当多个来源影响同一属性时：

1. **百分比类**：全部代数和，不分来源
2. **固定值类**：全部代数和，不分来源
3. **百分比 + 固定值混合**：先算百分比，再加固定值

```javascript
// 最终公式
final_ATK = Math.floor(base_ATK * (100 + totalPercentBonus) / 100) + totalFlatBonus;
```

## 五、控制效果定义

### 5.1 控制效果总表

| 控制类型 | 效果 | 可净化 | 可免疫 | 适用目标 |
|---------|------|--------|--------|---------|
| 眩晕 | 跳过整个回合（不行动、不普攻） | 是 | Boss 部分免疫 | 敌我双方 |
| 沉默 | 禁止使用主动技能（普攻可用） | 是 | 否 | 敌我双方 |
| 嘲讽 | 强制以嘲讽者为目标（仅限单体技能/普攻） | 是（驱散 buff） | Boss 部分/完全免疫 | 敌方 |
| 冰冻 | 跳过回合 + 受伤时提前解除 | 是 | Boss 部分免疫 | 敌方 |
| 减速 | SPD -X% | 是 | 否 | 敌我双方 |
| 加速 | SPD +X% | 否 | — | 友方 |

> 减速/加速技术上归类为属性类（stat），走 Step 0 SPD 修正逻辑，不参与控制互斥。
> 因其直接影响出手顺序，在设计角度可被视为控制手段，但实现时 `category = stat`，按属性类规则处理。

### 5.2 各控制效果详解

#### 眩晕

| 项目 | 内容 |
|------|------|
| 行为 | 目标在本回合内跳过行动（不普攻、不放技能、不触发"行动结束时"被动） |
| 持续 | 以轮为单位 |
| 与沉默的区别 | 眩晕完全跳过回合，沉默只是禁主动技（仍可普攻和触发被动） |
| 净化优先级 | 最高（技能系统 6.2：先解控） |

**眩晕时所受影响：**
- 不普攻 ✓
- 不放主动技 ✓
- 不触发被动（被动触发通常需要"行动完成"或"受到伤害"等条件，但像百战之躯这样在受伤时触发的被动不受眩晕影响）✓
- 受到伤害时的被动仍可触发（百战之躯、受到攻击的反击等）
- 被攻击/受伤不会解除眩晕（眩晕不是冰冻）

**眩晕回合的流程：**
```
轮到眩晕单位的回合 → 检查到有眩晕状态
  → 跳过该单位的所有行动
  → 不触发"回合开始时"效果（大部分不考虑） / 触发"回合开始时检测"效果（如减伤判定）
  → 直接进入下一个单位的回合
```

#### 沉默

| 项目 | 内容 |
|------|------|
| 行为 | 目标不能使用主动技能（有冷却的技能），但仍可使用普攻 |
| 持续 | 以轮为单位 |
| 净化优先级 | 高 |

**沉默时的限制：**
- 主动技能按钮灰化/不可点击
- 自动战斗 AI 跳过所有主动技能，视为"技能在冷却中"
- 普攻正常使用
- 被动技能正常触发

**沉默 vs 眩晕：**

| 维度 | 眩晕 | 沉默 |
|------|------|------|
| 普攻 | 跳过 | 可用 |
| 主动技 | 跳过 | 不可用 |
| 被动 | 受伤触发类可用，"行动结束时"类不可用 | 全部可用 |
| 控制强度 | 强（完全跳过） | 中（保留了普攻） |
| 适用场景 | 强控核心敌人 | 压制法系/治疗 |

#### 嘲讽

已在仇恨系统定义，此处简要归纳：

- 被嘲讽的敌人，其单体普攻和单体技能必须选择嘲讽者（玄甲）
- AOE 技能不受嘲讽影响
- Boss 部分免疫（持续减半）或完全免疫
- 嘲讽不是标准 debuff，而是一个独立的 `taunted` 标记
- 玄甲阵亡后嘲讽立即消失

#### 冰冻

| 项目 | 内容 |
|------|------|
| 行为 | 与眩晕类似（跳过回合），但受伤时提前解除 |
| 持续 | 1~2 轮（受伤提前解除） |
| 净化优先级 | 高 |
| 特殊 | Boss 部分免疫 |

**冰冻的特殊规则：**
- 冰冻状态下跳过回合（同眩晕）
- 如果冰冻目标受到任何伤害（哪怕是 1 点），冰冻立即解除
- 冰冻解除后，该角色本轮回合已被跳过，不再补动
- 冰冻体现在"控制有代价"：冰冻被伤害提前解除了，伤害方相当于"帮对方解控"

示例：
```
第 1 轮：方士冰冻敌人 A（持续 2 轮）
          惊鸿在冰冻后行动，打了敌人 A → 冰冻解除
          敌人 A 本回合已行动过了 → 冰冻解除但本轮回合也过了
第 2 轮：敌人 A 正常行动（冰冻已解）
```

#### 减速/加速

- 归类为状态类但不是"硬控制"
- 影响 SPD → 影响下一轮出手顺序
- 详细规则见回合流程文档第四章
- **减速/加速改变速度后，在当前轮已排好的序列中不生效，下一轮重建序列时生效**

#### 关于减速，补充已定稿的约定

来自技能系统 4.3 的注意事项：
> 减速对出手顺序的影响实时生效——如果减速施放在行动阶段中途，不会改变当前轮的出手顺序（已排序完了），但会影响下一轮的排序。

### 5.3 控制免疫与韧性（Boss 特殊规则）

> 普通敌人不受控制免疫影响，所有控制效果正常生效。下表仅适用于精英/Boss 级别的敌人。

| 敌人级别 | 嘲讽 | 眩晕 | 冰冻 | 沉默 | 减速 |
|---------|:----:|:----:|:----:|:----:|:----:|
| 精英敌人 | 完全 | 部分（持续减半） | 部分（持续减半） | 完全 | 完全 |
| 普通 Boss | 部分（持续 1 轮） | 部分（持续减半） | 部分（持续减半） | 完全 | 完全 |
| 高阶 Boss（5 章后） | 免疫 | 免疫 | 免疫 | 部分（持续减半） | 完全 |
| 世界 Boss | 免疫 | 免疫 | 免疫 | 免疫 | 部分（持续减半） |

**"部分免疫"的含义：**
- 控制效果仍然可以施加，但持续时间减半（向上取整）
- 例如：Boss 来源的眩晕持续 2 轮 → 对 Boss 实际持续 1 轮
- 如果原本持续 1 轮，减半后仍为 1 轮（向上取整）

**"完全免疫"的含义：**
- 该控制无法施加（免疫标记）
- 技能效果仍然生效（伤害、仇恨等），但控制效果不生效
- 如果用某个"带控制"的技能打完全免疫的 Boss：伤害照打，控制不生效

**韧性机制（预留扩展）：**
- 如需更精细的控制对抗，可引入"韧性值"系统：
  - 每次被控制时，目标积累韧性值
  - 韧性越高，后续控制持续时间越短
  - 战斗结束或长时间未受控后韧性衰减
- 当前阶段不需要此机制，用"部分免疫/完全免疫"即可

### 5.4 控制重叠规则

**同一个目标同时受到多个控制时：**

| 组合 | 处理方式 |
|------|---------|
| 眩晕 + 沉默 | 按眩晕处理（跳过行动，沉默的限制已被包含） |
| 冰冻 + 沉默 | 按冰冻处理（跳过行动，沉默不额外作用） |
| 眩晕 + 冰冻 | 按眩晕处理（冰冻的特点"受伤解除"仍然保留：眩晕中被攻击仍可解除冰冻） |
| 嘲讽 + 沉默/眩晕/冰冻 | 被嘲讽+被控：如果被控制跳过行动，嘲讽不生效（不能行动则不攻击） |

**控制覆盖（新控制替换旧控制）：**
- 新控制旧控制同时存在，各自独立
- 如果两个控制都限制了行动，取"限制更强的"实际生效
  - 眩晕 > 冰冻 > 沉默（强弱排序）
- 不同控制各自独立计时，各管各的到期移除

**例子：**

```
角色 A 被沉默（持续 2 轮）
第 1 轮：又被眩晕（持续 1 轮）
  → 本回合跳过行动（眩晕生效）
  → 沉默仍在计时
第 1 轮结束：眩晕剩余 0 → 移除；沉默剩余 1
第 2 轮：沉默仍生效 → 不能放主动技，但可以普攻
第 2 轮结束：沉默剩余 0 → 移除
第 3 轮：正常行动
```

## 六、DOT/HOT 机制

### 6.1 状态定义

| 类型 | 效果 | 触发时机 | 计算方式 |
|------|------|---------|---------|
| DOT | 每轮受到伤害 | 轮开始阶段（目标行动前） | 基于施法者 ATK × 固定倍率 |
| HOT | 每轮回复 HP | 轮开始阶段（目标行动前） | 基于施法者 ATK × 固定倍率 |

**为什么触发时机选"轮开始阶段"？**
- DOT 在目标行动前扣血 → 目标可能直接在行动前阵亡 → 跳过该目标的本轮行动（已阵亡无需行动）
- HOT 在目标行动前回血 → 目标以更健康的状态进入本轮的回合
- 两者都在轮开始时结算，意味着 DOT/HOT 在同一轮内只触发 1 次

> 注意：轮开始阶段先触发 DOT/HOT，然后再进入行动阶段。如果 DOT 导致某目标阵亡，该目标在行动阶段被跳过。

### 6.2 伤害/治疗公式

**DOT 伤害公式：**

```
dot_damage = caster_ATK × dot_multiplier / 100
```

- 走**简化**的伤害流水线（不经过防御减免、暴击、等级补正、随机波动）
- 仅受 Step 7 影响（"造成的伤害 +X%" 适用，"受到的伤害 +X%" 适用）
- 不受暴击影响
- 结果向下取整，最少 1 点

```javascript
function calcDotDamage(caster, target, dotMult) {
  let dmg = Math.floor(caster.atk * dotMult / 100);
  // Step 7: 增伤/易伤修正
  let bonus = caster.dmgDealtBonus + target.dmgTakenBonus;
  // dmgTakenBonus: 正值=易伤(+X%)，负值=减伤(-X%)
  dmg = Math.floor(dmg * (100 + bonus) / 100);
  return Math.max(1, dmg);
}
```

**HOT 治疗公式：**

```
hot_heal = caster_ATK × hot_multiplier / 100
```

- 走简化治疗流水线（不经过防御、暴击）
- 受"治疗效果 +X%"修正
- 结果向下取整，最少 1 点

```javascript
function calcHotHeal(caster, hotMult) {
  let heal = Math.floor(caster.atk * hotMult / 100);
  heal = Math.floor(heal * (100 + caster.healBonus) / 100);
  return Math.max(1, heal);
}
```

### 6.3 DOT/HOT 叠加规则

**同类型不同层：**

```
每层 DOT 独立记录，不合并
例如：敌人 A 挂 2 层灼烧（分别来自方士和装备），各层独立计算伤害
```

**层数上限：** 每个角色身上的 DOT 最多 3 层（相同 ID 的），HOT 最多 3 层（相同 ID 的）

**同 ID 刷新：** 使用同一技能重新施加时，重置层数到 1，刷新持续时间

**为什么要分层？**
- 不同来源的 DOT 由不同角色施放，各自计算伤害，清晰可追踪
- 分层叠加上限 3 层，避免无限叠层导致伤害失控
- 但同一来源不叠层（同 ID 刷新）

### 6.4 DOT 能暴击吗？

**规则：DOT 不能暴击。** 理由：
- DOT 是持续性伤害，如果还能暴击，期望收益过高
- DOT 本身的优势是"稳定每轮触发"，和暴击的"不稳定高爆发"是两套体系
- 如果后续需要 DOT 暴击，可通过装备词条或被动解锁

**例外记录：** 技能系统 12.4 中配置表已有 `治疗暴击` 的待定事项。DOT 暴击同理，当前不开放。

### 6.5 多 DOT 叠加计算示例

```
敌人 A 被挂 2 层 DOT：
  层 1：方士的灼烧，ATK=500，倍率=20%，dot_damage=100
  层 2：装备灼烧，ATK=60（固定值），倍率=20%，dot_damage=12

轮开始阶段：
  敌人 A 受到 100 + 12 = 112 伤害（防御不减免，无暴击）
  敌人 A 进入行动阶段时 HP 已减少 112
```

## 七、净化和驱散

### 7.1 可净化 vs 不可净化

**规则（来自技能系统 6.2，正式化）：**

- **可净化：** 所有标记为 `cleanseable: true` 的 debuff
- **不可净化：**
  - 所有 buff（来自己方的正效果）
  - 标记为 `cleanseable: false` 的特殊 debuff（如 Boss 专属标记）
  - 嘲讽（不是 debuff，是独立标记）
  - 护盾（是 buff，不可净化——但可被驱散）

**当前已有效果的可净化性：**

| 效果 | 标记 | 可净化？ | 备注 |
|------|------|---------|------|
| DEF+20%（铁壁嘲讽） | buff | 不可（是 buff） | — |
| ATK+30%（鹰眼） | buff | 不可（是 buff） | 技能系统 5.3 已注明 |
| SPD-20%（缚灵咒） | debuff | 可 | 技能系统 4.3 已注明 |
| 护盾 | buff | 不可净化（但可驱散） | 7.3 节详述 |
| 眩晕 | debuff | 可 | — |
| 沉默 | debuff | 可 | — |
| 冰冻 | debuff | 可 | — |
| 嘲讽 | 独立标记 | 不可 | 非 debuff |
| DOT | debuff | 可 | 净化移除整层 DOT |
| Boss 专属标记 | debuff | 不可 | 在具体技能中标注 `[不可净化]` |

### 7.2 净化优先级（来自技能系统 6.2，正式化）

素问回春术净化时，按以下优先级选择目标身上的 debuff：

```
优先级 1：控制类 debuff（眩晕/沉默/冰冻）—— 如果能动，先解控
优先级 2：DOT/生存威胁类 debuff —— 掉血/易伤会致死
优先级 3：属性减益类 debuff（ATK-/DEF-/SPD-）—— 最后才解这些
```

- 一次净化只清除 1 个 debuff
- 如果同优先级有多个 debuff，按施加时间从新到旧选择
- 如果目标身上没有可净化的 debuff，净化效果不触发（但治疗正常执行，仇恨照常产生）

### 7.3 驱散敌方 Buff

**驱散（Dispel）和净化（Cleanse）的区别：**

| 概念 | 目标 | 效果类型 |
|------|------|---------|
| 净化（Cleanse） | 友方 | 移除 debuff |
| 驱散（Dispel） | 敌方 | 移除 buff |

**当前游戏已有的驱散相关设计：**
- 目前没有专门驱散敌方 buff 的技能（技能系统表中无此类技能）
- 护盾可以被驱散（技能系统 7.1 规则 4），但当前阶段没有驱散护盾的手段

**驱散规则（预留）：**
- 驱散移除敌方身上指定数量的 buff
- 护盾有独立的 buff ID（`BUFF_SHIELD`），驱散时可以指定"优先驱散护盾"或"随机驱散 1 个 buff"
- 某些 buff 可标记为 `undispellable`（不可驱散），如 Boss 的专属增益

**驱散优先级（未来扩展）：**
```
优先级 1：护盾类 buff（BUFF_SHIELD）
优先级 2：攻击性增益（ATK+/CRIT+ 等）
优先级 3：防御性增益（DEF+ 等）
```

## 八、与已定稿设计的对接

### 8.1 与伤害公式 Step 7 匹配

伤害公式 Step 7：

```
total_bonus = sum(attacker_dmg_bonus) + sum(defender_dmg_taken_bonus)
reduced_damage = reduced_damage × (100 + total_bonus) / 100
```

**哪些 Buff/Debuff 影响 Step 7：**

| 效果类型 | 影响哪个变量 | 例子 |
|---------|------------|------|
| "造成的伤害 +X%" | `attacker_dmg_bonus` | 某个 buff：增伤 +15% |
| "造成的伤害 -X%" | `attacker_dmg_bonus`（负值） | 某个 debuff：虚弱 -10% |
| "受到的伤害 +X%"（易伤） | `defender_dmg_taken_bonus` | 某个 debuff：破绽 +20% |
| "受到的伤害 -X%"（减伤） | `defender_dmg_taken_bonus`（负值） | 百战之躯 -6% |

**聚合规则：** 所有影响 Step 7 的修正，代数和叠加。

```javascript
// 例如：攻击者有 +15% 增伤 buff，防御者有 -10% 减伤和 +20% 易伤
total_bonus = (+15) + (-10) + (+20) = +25
// 最终伤害 × 125%
```

### 8.2 与回合流程各阶段匹配

| 阶段 | Buff/Debuff 相关动作 |
|------|-------------------|
| **轮开始** | HOT/DOT 结算 → 治疗/伤害数值应用 → 如阵亡标记 |
| **行动阶段 | 每个单位回合： |
| | └ 回合开始 → 触发性效果（如"回合开始时解控"） |
| | └ 选择行动 → 检查控制状态（眩晕跳过、沉默禁用主动） |
| | └ 执行行动 → 可能产生新的 buff/debuff |
| | └ 回合结束 → 触发"回合结束时"效果（悬壶等） |
| **轮结束** | 1. 触发"每轮结束时"效果 |
| | 2. Buff/Debuff 计时 -1 |
| | 3. 移除到期效果（剩余轮数 ≤ 0） |
| | 4. 胜负判定 |

**重要：** DOT/HOT 在轮开始阶段结算，意味着：
- DOT 生效 → 目标掉血 → 可能阵亡 → 跳过该目标的本轮行动
- HOT 生效 → 目标回血 → 以更好的状态进入行动阶段
- DOT/HOT 在当前轮的"轮结束"阶段不再结算（只触发一次）

### 8.3 与仇恨系统的交互

| 行为 | 仇恨值 | 参考 |
|------|--------|------|
| 施加 Buff（对友方） | 30（对全体敌人） | 仇恨系统 2.1 |
| 施加 Debuff（对敌方） | 30（对全体敌人） | 仇恨系统 2.1 |
| 施加护盾 | `shield × 0.3`（对全体敌人） + 30 固定（施加增益） | 仇恨系统 2.1 |
| DOT 每轮伤害 | 不产生新仇恨 | DOT 是已有技能的持续效果，仇恨已在技能命中时产生 |
| HOT 每轮治疗 | 不产生新仇恨 | 同上 |
| 净化 | 30（对全体敌人） | 技能系统 6.2 |
| 驱散 | 30（对全体敌人） | 预留规则 |

**设计理由：** DOT/HOT 不产生每轮仇恨，是因为仇恨已在施放技能那一刻产生。如果一个 DOT 持续 3 轮，每轮都产生仇恨，等于 1 个技能产生 4 次仇恨，显然不合理。

### 8.4 与技能系统所有 Buff/Debuff 引用的双向一致性检查

| 技能系统中的引用 | 本方案覆盖情况 |
|----------------|--------------|
| BUFF_XJ_DEF_UP（铁壁 DEF+20%） | 属性类 buff，refresh 规则，可净化=false |
| BUFF_JH_HAWK_EYE（鹰眼 ATK+30%, CRIT+20%） | 属性类 buff（两个属性绑定同一个 ID），refresh 规则，可净化=false |
| DEBUFF_SPD_DOWN（缚灵咒 SPD-20%） | 属性类 debuff，refresh 规则，可净化=true |
| BUFF_SHIELD（护盾） | 状态类，replace 规则，可净化=false |
| 嘲讽（taunted 标记） | 状态类（独立标记），特殊规则 |
| 净化（回春术） | 按 7.2 优先级执行 |
| 百战之躯（被动减伤） | 在 Step 7 以降低 `defender_dmg_taken_bonus` 形式实现，按 4.2 规则 |
| 控制类（后续技能可能引入） | 本方案第 5 章已定义框架 |

## 九、边界情况

### 9.1 Buff 在角色死亡后是否移除

**规则：目标死亡后，其身上的所有 buff/debuff 立即移除。**

理由：
- 死人不应该有 buff（符合直觉）
- DOT/HOT 在死者身上无意义
- 防止复活后带有死亡前的 buff 导致异常

**特别注意：施放者死亡不影响其施加的 buff（2.4 节已规定），但目标死亡肯定移除。**

### 9.2 控制重叠

见 5.4 节。

**补充场景：嘲讽 + 被眩晕同时发生**

```
玄甲嘲讽了敌人 A（持续 2 轮）
在第 2 轮，敌人 A 被眩晕（持续 1 轮）
→ 敌人 A 本回合跳过行动
→ 嘲讽不生效（不能行动则不攻击）
→ 嘲讽仍在计时
→ 嘲讽持续轮数 -1（不受眩晕影响）
```

### 9.3 Debuff 命中率与必中

**规则：本游戏所有技能和效果为必中，没有命中率/闪避系统。**

理由：
- 不需要引入命中和闪避属性（节省属性维度）
- 控制效果的成功与否取决于目标是否免疫，而不是随机数
- 玩家能清晰预判"这个技能放出去就一定会生效"

**唯一的随机性是暴击判定（伤害公式 Step 5）和伤害波动（Step 6）。**

### 9.4 Buff 持续轮数计算（施加轮算不算？）

来自回合流程第四章，此处再次确认：

**规则：施加轮算作第 1 轮，轮末减 1。**

```
第 1 轮 行动阶段：施加 buff（剩余轮数 = 2）
第 1 轮 轮结束：剩余轮数 -1 → 剩余 1
第 2 轮 行动阶段：buff 生效
第 2 轮 轮结束：剩余轮数 -1 → 剩余 0 → 移除
第 3 轮 行动阶段：buff 已消失
```

**"持续 1 轮"的含义：**
- 施加轮起作用，轮末减到 0 移除
- 实际覆盖时间：从施加时刻到同一轮的轮结束
- 如果是行动阶段后期施加的"持续 1 轮"，可能只在轮末前生效一小段时间
- 但因为是回合制，轮末才移除，总归覆盖到当前轮结束

### 9.5 新控制覆盖旧控制

见 5.4 节。新控制和旧控制各自独立计时，互不覆盖（除非是同 ID 的同种控制）。

**同 ID 的相同控制：** 刷新持续时间（refresh 规则）
- 新眩晕覆盖旧眩晕 → 重置持续轮数
- 新沉默覆盖旧沉默 → 重置持续轮数

**不同 ID 的不同控制：** 各自独立
- 眩晕 + 沉默可以同时存在（虽然眩晕的效果覆盖了沉默）
- 各自独立计时

### 9.6 DOT 在目标死亡时的处理

- 如果 DOT 在轮开始阶段导致目标阵亡 → 目标从行动阶段中移除
- 目标死亡后，所有 DOT 层数随目标死亡清除
- 同一 DOT 不会"转移到"其他目标

### 9.7 净化的边界

| 场景 | 处理 |
|------|------|
| 目标无 debuff，使用回春术 | 净化不触发，治疗正常执行，仇恨 30 仍产生 |
| 目标有 2 个 debuff，净化 1 个 | 按优先级选 1 个移除，另一个保留 |
| 目标有不可净化的 debuff | 净化跳过不可净化项，选下一个可净化的 |
| 目标所有 debuff 都不可净化 | 净化不触发（同"目标无 debuff"） |

### 9.8 行动阶段内施加的 Buff 对当前轮的影响

- **属性类（ATK/DEF/SPD/CRIT 等）**：立即影响当前轮后续行动
  - 如果方士在序列中比惊鸿先动，给敌人上 SPD-20%，当前轮的出手序列已固定，减速**不影响当前轮**（下一轮生效）
  - 如果方士给惊鸿上 ATK+30% 的 buff，惊鸿在本轮还没行动，则惊鸿行动时 ATK+30% 已生效
- **控制类**：立即生效，从当前单位开始
  - 如果方士眩晕了还没行动的敌人，该敌人在当前轮跳过行动
- **DOT**：本轮不触发（已经过了轮开始阶段），下一轮开始阶段触发

## 十、UI/实现注意

### 10.1 Buff/Debuff 图标数量上限

**规则：每个角色头像旁最多显示 6 个 Buff/Debuff 图标。**

处理逻辑：
1. 优先显示**控制类**效果（眩晕、沉默等——这些对决策最重要）
2. 其次显示**持续类**效果（DOT/HOT——玩家需要知道自己在掉血/回血）
3. 再显示**属性类**效果（ATK+/DEF+ 等）
4. 超过 6 个时，多余的效果**不在 UI 显示**但仍在后台生效
5. 在角色详情面板中可查看全部生效的 buff/debuff 列表

**为什么是 6 个？** 微信小程序的屏幕空间有限，角色头像旁的空间放 6 个小图标已经是上限。

**实现方式：**

```javascript
const MAX_VISIBLE_BUFFS = 6;
const DISPLAY_PRIORITY = ['control', 'dot', 'hot', 'status', 'stat'];

function getVisibleBuffs(unit) {
  const sorted = unit.activeEffects.sort((a, b) => {
    return DISPLAY_PRIORITY.indexOf(a.category) - DISPLAY_PRIORITY.indexOf(b.category);
  });
  return sorted.slice(0, MAX_VISIBLE_BUFFS);
}
```

### 10.2 剩余轮数显示

- 每个 Buff/Debuff 图标左下角显示剩余轮数（数字）
- 剩余 1 轮时：用特殊颜色（如红色/橙色）警示
- 持续到战斗结束（-1）：显示为 ∞ 符号

### 10.5 性能关注点

- **每轮轮末的操作**：所有效果计时减 1 + 到期移除 = O(n) 其中 n = 场上所有效果实例数
  - 3v6 战斗，每个单位身上平均 3 个效果，约 27 个效果实例
  - 整数运算，性能无压力
- **DOT/HOT 结算**：轮开始阶段遍历所有存活单位，检查 DOT/HOT 效果
  - 最多 9 个单位 × 6 层 DOT = 54 次计算
  - 每次计算是简单的整数乘除法
- **Step 0 面板计算**：每次伤害前计算（不是缓存），所以 buff 变化即时生效
  - 需要遍历目标身上的所有 stat 类效果
  - 建议每次结算时实时计算，不缓存面板（避免缓存失效问题）
