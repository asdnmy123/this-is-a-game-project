## 方案名称：令行禁止

### 核心规则

1. 每轮分三个阶段：轮开始 → 行动阶段 → 轮结束
2. 所有存活单位按速度降序出手，每人每轮只动一次
3. 速度只决定顺序，不决定行动次数
4. Buff/Debuff 持续以"轮"计算，在轮结束时统一计次
5. 普通战斗 20 轮上限，Boss 战 30 轮上限

---

### 详细设计

#### 一、回合阶段划分

每轮严格按以下三个阶段顺序执行：

**阶段一：轮开始**
1. 胜负判定（兜底，捕捉轮末判定后遗漏的极端情况）
2. 触发"每轮开始时"效果（DOT/HOT 结算、被动技触发等）
3. 击杀事件触发（通灵/追猎）
4. 胜负判定（捕捉 DOT/HOT 杀敌情况）
5. 重置"本轮回合是否使用过"标记（仅用于限制"每轮一次"的技能）
6. 各种领域/结界持续减 1（如有）

**阶段二：行动阶段**
1. 本阶段生成**出手序列**：所有存活单位按速度从高到低排序
2. 按序列依次执行每个单位的回合
3. 一个单位的回合内流程：
   a. **回合开始** — 触发"回合开始时"效果（如自动回血、解除睡眠等）
   b. **选择行动** — 普攻 / 技能 / 防御 / 道具（自动战斗时 AI 决定）
   c. **执行行动** — 动画 + 数值结算
   d. **回合结束** — 触发"回合结束时"效果（如反击判定等）
4. 行动中新产生的单位（如召唤物）在本轮不再行动，排入下一轮序列

**阶段三：轮结束**
1. 触发"每轮结束时"效果
2. Buff/Debuff 计时统一减 1（规则见下文第四章）
3. 持续时间为 0 的效果被移除
4. 胜负判定：若一方全灭 → 战斗结束
5. 检查回合上限（第六章）：若达到上限 → 按规则结算

> 注意：轮结束的 Buff 减1 发生在胜负判定之前。"持续1轮"的 buff 从施加时生效，持续到当前轮结束时统一减 1 归零并移除。也就是说它覆盖的是"从施加减到轮末"这段时间，不是整轮从头到尾。

#### 二、速度与出手顺序

**速度排序规则：**

- 出手序列 = 所有存活单位按「速度」属性降序排列
- 速度值相同时的优先级：
  1. 玩家方优先于敌方
  2. 同为玩家方：位置 1 > 2 > 3
  3. 同为敌方：位置 1 > 2 > 3 > …（按敌方编队序号）
- 出手序列在每轮行动阶段开始时**重新生成**，不受上一轮影响

**速度在战斗中的作用：**

- 核心作用：决定出手顺序，速度高的单位先行动
- 先手价值：
  - 玄甲（坦）：先手嘲讽 + 建立仇恨，抢在敌人 dps 出手前拉住怪
  - 方士（法）：先手控场/上 debuff，限制敌人行动
  - 惊鸿（弓）：先手点杀脆皮，造成人数优势
  - 素问（奶）：先手上盾/加血，保护队友
- 速度不加行动次数，不加伤害，不加治疗量

**为什么不让高速单位多动：**

> 简单类比：速度快的角色不是跑两趟，而是跑在队友前面。
> 多动系统（如崩铁）需要完整的行动值/行动条底层来支撑，对于小队 3v6 的规模来说收益不高，反而增加平衡难度。
> 一个角色一回合动一次，玩家算得清、看得懂。

---

#### 三、先手与速度优势

**先手的战略优势：**
- 抢先控场：方士先手冰冻/减速高威胁敌人
- 抢先加盾：素问先手给全队套护盾
- 抢先减员：惊鸿先手秒掉敌方脆皮输出
- 抢先上嘲讽：玄甲先手建立仇恨

**慢速的后发优势（对冲先手）：**
- 后手可以看情况应对 — 比如等对方上了 buff 后再驱散
- 在自己行动后才开始计算 buff 持续时间（见第四章），所以慢速角色的 buff 实际覆盖的"真实时间"更长？不，本文按轮计算，走自己的行动顺序不影响持续时间。

**先手硬保底：**
- 为了保证战斗体验，建议给玩家方一个硬保底机制：
  - 出手序列中，玩家方**至少有一个角色**在敌方出手前行动
  - 如果玩家方所有单位都排在敌方之后，把玩家方速度最快的单位提到敌方第一个单位之前（保持它在玩家方内部的相对顺序不变）
  - 目的：防止玩家被敌方全体先手打一套还没动就减员（mobile 游戏体验很差）
- 设计理由：这是单机游戏，不是竞技游戏，不需要绝对公平。玩家的体验优先于竞技公平。

> 或者说，这个保底意味着：第一轮玩家方必然有人能动。不会出现"全员还没动就被秒了"的情况。

#### 四、Buff/Debuff 持续回合计算

**核心规则：**

1. 持续时间以 **「轮」** 为单位
2. 每个效果有 `剩余轮数` 计数器，初始值 = 技能描述的持续轮数
3. **在每轮结束阶段**，所有效果的剩余轮数统一减 1
4. 剩余轮数降至 0 后立即移除

**例子：**

> 素问在第二轮的行动中给自己上了「攻击强化·持续2轮」：
> - 第二轮结束 → 剩余轮数：1
> - 第三轮结束 → 剩余轮数：0，效果移除
> - 该 buff 实际覆盖了第二轮剩余时间 + 整个第三轮

**特殊情况：**

- **关于"持续到战斗结束"**：设置剩余轮数为 `-1`（特殊标记），不减
- **叠加与刷新**：
  - 同名的 buff/debuff 叠加时，刷新持续时间为**较大值**
  - 例如：已有持续 2 轮的 buff（剩余 1 轮），重新施加持续 3 轮的版本 → 剩余变为 3
  - **不合并层数**（除非明确设计为可叠层，如 debuff 层数系统 — 待定）
- **谁的效果消失**：
  - buff 施放者阵亡 → buff 是否消失？两种处理：
    - **方案 A（推荐）**：施放者阵亡后 buff 保留（简单，避免玩家困惑"我刚加的 buff 怎么没了"）
    - **方案 B**：施放者阵亡后 buff 消失（更真实，但复杂）
  - 推荐方案 A，等后续需要再改

---

#### 五、第一轮特殊规则

**第一轮走正常规则：**

- 速度排序 → 行动 → 轮结束，和普通轮次一样
- 没有"先手必定是我方"的硬性设定
- 先手软保底机制见第三章

#### 六、回合上限

| 战斗类型 | 上限 | 到达上限后的处理 |
|---------|------|----------------|
| 普通战斗 | 20 轮 | 判定玩家方失败（撤退不计入失败次数？待定） |
| Boss 战 | 30 轮 | Boss 进入**「狂暴」**状态：刚体（DEF+50%）+ 怒意（造成伤害+50%，每额外一轮 +10%）|
| 教学关/剧情关 | 无上限 | 保证玩家能过 |

**设计理由：**
- 移动端游戏单局时间不宜过长，20 轮足够决定胜负
- 回合上限逼着玩家提升练度，而不是靠"刮痧磨过去"
- Boss 狂暴给玩家一种"得抓紧输出"的紧张感
- 教学关无上限防止卡关

### 设计理由

**为什么采用"速度排序，每人一动"而不是行动值系统：**

1. **学习成本低**：一眼看懂，不用理解"行动条""行动值""提前度"这些概念
2. **实现成本低**：就是排序，不需要时间轴推进的 tick、不需要预测谁下一个动
3. **调试容易**：谁什么时候动是确定的，复现 bug 简单
4. **适合小队规模**：3v6 = 9 个单位，排序一次后依次执行，性能开销忽略不计
5. **策略深度足够**：速度影响 turn order 已经是成熟的策略维度（参考八方旅人、女神异闻录）

**为什么持续回合按"轮"算而不是按"行动次数"算：**
- 按行动算需要精确追踪每个效果被施加后经过了几个单位行动，实现复杂
- 按轮算简单粗暴：轮结束统一减 1，容易 debug
- 对于 3v6 = 9 个单位的规模，按行动算和按轮算差别不大

**为什么给玩家方保底：**
- 这是单机游戏，不是竞技对战。玩家的爽感 > 公平性
- 防止"全队没动就团灭"的极端情况（速度被碾压 + 对面正好全是 AOE）
- 保底只在速度和硬件层面做一个软限制，不会显著影响正常战斗

### 实现注意

**伪代码示意：**

```python
# 每轮主循环
def run_round():
    # 阶段一：轮开始
    if check_win_loss(): return           # 第1次判定：兜底
    run_round_start_effects()              # DOT/HOT 结算
    trigger_kill_events()                  # 击杀事件触发
    if check_win_loss(): return           # 第2次判定：捕捉 DOT 杀敌
    reset_per_round_flags()                # 重置"每轮一次"标记
    tick_arena_timers()                    # 领域/结界计时
    
    # 阶段二：行动阶段
    turn_order = sort_by_speed(all_alive_units)
    # 玩家方至少有一位在敌人之前行动（软保底）
    apply_first_turn_bonus(turn_order)
    
    for unit in turn_order:
        if not unit.is_alive:            # ← 防止行动中死亡的单位继续执行
            continue                     # ←
        run_turn(unit)
        if check_win_loss(): return
    
    # 阶段三：轮结束
    run_round_end_effects()
    tick_all_buff_durations()
    remove_expired_effects()
    if check_win_loss(): return
    
    # 检查回合上限
    if current_round >= max_rounds:
        handle_round_limit()
        return

# 一个单位的回合
def run_turn(unit):
    trigger_on_turn_start(unit)
    action = choose_action(unit)
    execute_action(unit, action)
    trigger_on_turn_end(unit)

# 速度排序（含软保底）
def sort_by_speed(units):
    # 正常速度降序
    sorted_units = sorted(units, key=lambda u: u.speed, reverse=True)
    # 检查玩家方是否有人在敌方之前动
    # 如果没有，找到玩家方速度最快的，提前到敌方最慢的之前
    return apply_first_turn_guarantee(sorted_units)
```

**需要定义的数据结构：**

```python
class BattleState:
    round_number: int           # 当前轮数（从 1 开始）
    max_rounds: int             # 当前战斗最大轮数（普通=20，Boss=30）
    phase: str                  # "ROUND_START" | "ACTION" | "ROUND_END"
    turn_order: List[Unit]     # 当前轮出手序列
    current_turn_index: int     # 当前是第几个行动
    units: List[Unit]          # 所有参战单位
    effects: List[Effect]      # 所有活跃的 Buff/Debuff

class Effect:
    source: Unit               # 施放者
    target: Unit               # 目标（每个 Effect 实例只绑定一个目标）
    effect_type: str           # "BUFF" | "DEBUFF"
    effect_id: str             # buff 唯一标识（用于刷新判定）
    remaining_rounds: int      # 剩余轮数，-1 = 持续到战斗结束
    stats_modifiers: dict      # 属性修正值

# 群体效果：每个目标各自生成一个独立的 Effect 实例
# 例如素问对全队加攻，生成 3 个 Effect，每个 target 指向一个队友
```

**软保底实现细节：**

```python
def apply_first_turn_guarantee(units_sorted):
    # 找出第一个玩家单位和第一个敌方单位的 index
    # 用 None 兜底，防止一方全灭时 next() 抛异常
    first_player_idx = next((i for i, u in enumerate(units_sorted) if u.is_player), None)
    first_enemy_idx = next((i for i, u in enumerate(units_sorted) if not u.is_player), None)
    
    # 如果某一方已经全灭，不需要调整
    if first_player_idx is None or first_enemy_idx is None:
        return units_sorted
    
    # 如果玩家方所有单位都在敌方之后 → 上提
    if first_player_idx > first_enemy_idx:
        # 找到玩家方速度最快的单位
        fastest_player = max([u for u in units_sorted if u.is_player],
                             key=lambda u: u.speed)
        # 把它提到敌方第一个单位之前（但保持它在玩家方内部的相对顺序）
        units_sorted.remove(fastest_player)
        units_sorted.insert(first_enemy_idx, fastest_player)
    
    return units_sorted
```

**需要关注的点：**
- 速度 buff 改变单位速度 → 出手序列**立即**在下一轮重建，不影响当前轮已排好的序列
- 单位死亡/复活 → 出手序列每轮重建，死亡单位自动跳过，复活后正常排序
- 速度排序先于行动执行，不要一边行动一边排序
