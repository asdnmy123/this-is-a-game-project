## 一、胜负条件

### 1.1 胜利条件

**标准规则：全部敌方单位 HP ≤ 0 时，玩家方胜利。**

补充：
- 战斗胜利只需敌方全灭，与己方存活人数无关。即使己方只剩 1 人，只要杀光了对面就算赢。
- 胜利不计入"需要消灭的敌人数量阈值"——全部击杀即可，不要求全清再补一刀。
- 召唤物/临时单位也算敌方单位，必须全部消灭才算胜利。

### 1.2 失败条件

**标准规则：全部己方单位 HP ≤ 0 时，玩家方失败（战斗不能/团灭）。**

补充：
- 不需要所有己方单位同时倒地。只要最后一个存活单位的 HP 降到 0，战斗立即判定失败。
- 己方单位全部阵亡后，战斗不再继续（不会出现"死人被鞭尸"的情况）。

### 1.3 阵亡定义

**HP = 0 即阵亡。没有倒地倒计时、没有濒死抢救窗口。**

理由：
- 倒地倒计时系统（如 XCOM、火焰纹章）需要额外机制（战场急救、倒下后还能被攻击），对移动端单机小游戏的复杂度收益不高。
- 单局最多 9 个单位，阵亡判定简单直接：HP=0 → 移除行动序列 → 不可被治疗选中。
- 角色死后不做"假死"标记——就是死了，战斗结束前不会再活过来（除非有复活技能——当前未设计，预留扩展）。

### 1.4 平局/超时机制

参考回合流程文档第六章，此处在轮末判定部分正式化：

| 战斗类型 | 回合上限 | 到达上限后的处理 |
|---------|:-------:|----------------|
| 普通战斗 | 20 轮 | 判定玩家方失败（撤退处理，见 3.2 节） |
| Boss 战 | 30 轮 | Boss 进入**「狂暴」**状态：获得「刚体」（DEF +50%）+「怒意」（造成伤害 +50%，每额外一轮再 +10%） |
| 教学关/剧情关 | 无上限 | 保证玩家能过 |

**为什么普通战斗超时判玩家负而不是按剩余 HP 比？**

两种方案的对比：

| 方案 | 处理 | 理由 |
|------|------|------|
| A（本方案） | 判玩家负 | 简单明确，没有模糊地带。鼓励玩家提升练度而不是刮痧。20 轮打不赢说明练度不够或配队有问题。 |
| B | 按剩余 HP 总和比 | 需要精确计算双方残存 HP 总量，可能出现"17% vs 15% 玩家险胜"的尴尬局面。玩家会觉得"差一点就赢了"的挫败感大于成就感。 |

**Boss 狂暴的细节：**

狂暴由两个独立 buff 组成：

1. **刚体**：防御力（DEF）+50%。效果不叠加，不随轮数增长。
2. **怒意**：造成的伤害 +50%。每额外增加一轮战斗，怒意层数 +1，每层再增加 10% 伤害（即第 31 轮 = +60%、第 32 轮 = +70%……）。

其他规则：
- Boss 达到 30 轮上限时触发狂暴，不是"战斗结束"——Boss 战超时不判负，而是大幅提升难度
- 如果玩家在狂暴状态下全灭 → 判负（正常团灭流程）
- 狂暴状态持续到战斗结束（玩家败或 Boss 亡）
- 狂暴是永久性的（不会几轮后消退），逼玩家在有限回合内结束战斗
- 刚体和怒意不占用常规 buff 位，不可被驱散
- 具体数值由数值策划调优

### 1.5 长时战斗的容错性设计

虽然回合上限已经限制了对局时长，但仍然要考虑极端情况：

- **双方都只剩奶妈（素问 vs 治疗型敌人）** → 双方都打不死对方，直到普通战斗 20 轮上限判玩家负
- **玄甲只剩 1 人 vs 多个敌人** → 玄甲输出低，靠百战之躯硬扛，但最终会因为输出不够而超时
- **玩家只剩惊鸿 vs Boss** → 惊鸿高输出但脆皮，有可能翻盘，但风险高

这些"苟活"局面不需要额外干预，回合上限自然终结。

## 二、胜负判定时机

### 2.1 判定锚点

胜负判定在回合流程中的位置，正式化如下：

```
轮开始阶段：
  1. 胜负判定 ◄── 第 1 次判定（兜底，捕捉轮末判定后遗漏的极端情况）
  2. DOT/HOT 结算
  3. 击杀事件触发（通灵/追猎）
  4. 胜负判定 ◄── 第 2 次判定（捕捉 DOT 杀敌情况）
  5. 重置"每轮一次"标记
  6. 领域/结界计时

行动阶段：
  1. 生成出手序列
  2. 依次执行每个单位的回合：
     └─ 每次造成伤害后 → 检查目标是否阵亡
        └─ 如果阵亡 → 触发击杀事件（通灵/追猎）
           └─ 击杀事件处理完后 → 胜负判定 ◄── 第 3 次判定
     └─ 追击伤害后同样执行上述检查

轮结束阶段：
  1. 触发"每轮结束时"效果
  2. 仇恨衰减 ×80%
  3. Buff/Debuff 计时 -1
  4. 移除到期效果
  5. 胜负判定 ◄── 第 4 次判定
  6. 回合上限检查（如果达到上限 → 按 1.4 处理）
```

### 2.2 四次判定锚点的分工

| 锚点 | 触发时机 | 作用 |
|------|---------|------|
| 第 1 次（轮开始·前置） | 每轮最开头 | 兜底，捕捉轮末判定后遗漏的极端情况 |
| 第 2 次（轮开始·DOT后） | DOT/HOT 结算 + 击杀事件后 | 捕捉 DOT/HOT 杀敌情况 |
| 第 3 次（行动中） | 每次伤害/治疗后 | **主要判定点**。绝大多数战斗在此结束 |
| 第 4 次（轮结束） | 轮末效果全部处理完 | 兜底判定。处理轮末被动触发导致的阵亡 |

### 2.3 判定流程伪代码

```javascript
function checkBattleEnd(battle) {
  const allEnemiesDead = battle.enemies.every(e => !e.isAlive);
  const allPlayersDead = battle.players.every(p => !p.isAlive);

  if (allEnemiesDead) {
    return 'victory';
  }
  if (allPlayersDead) {
    return 'defeat';
  }
  return 'continue'; // 战斗继续
}
```

### 2.4 双方同时阵亡的处理

**核心原则：「死亡即停」。** 单位阵亡后，其死亡触发的效果不再执行。玩家击杀最后一个敌人 = 立即胜利，不处理敌人死亡后的任何效果。

例外：自爆/自损伤害在技能执行时已产生，自爆者虽死但伤害已发出，仍然生效。

#### 场景一：玩家击杀最后一名敌人（标准胜利）

玩家行动击杀敌方最后一人 → 战斗立即判定玩家胜利。敌人的濒死反扑/死亡触发效果不再执行。

```
示例：场上只剩惊鸿（HP 200）vs 敌人头目（HP 100）
惊鸿行动，破甲箭打头目 → 头目 HP 归零
→ 立即胜利！战斗结束。
→ 敌人的"濒死反扑"不再触发。
```

#### 场景二：敌方自爆/反伤杀死己方全体

自爆伤害在执行时已产生，自爆者死亡不影响已发出的伤害：

1. 自爆伤害结算 → 己方全体阵亡
2. 自爆者自身 HP ≤ 0
3. 双方全灭 → 判玩家负

```
示例：敌人头目发动自爆"对全体造成 ATK×300% 伤害，自身受到 9999 伤害"
1. 自爆伤害发出 → 对惊鸿造成 300 伤害
2. 头目自身 HP 归零（自爆伤害）
3. 惊鸿 HP 归零
4. 双方全灭 → 判玩家负
```

#### 场景三：同一结算点同时阵亡（平局）

当双方在**同一个结算点**同时阵亡（例如双方 DOT 在同一轮开始阶段同时跳死最后一人），此时不存在"谁先杀谁"的时序差异：

→ 判**平局**。体力退还，不计入失败次数，允许重试。

**平局处理流程：**
```
判定平局 → 立即停止行动
    ↓
显示"势均力敌"
    ↓
体力退还（返还进入战斗时扣除的体力）
    ↓
不计入失败次数，不影响星级评价
    ↓
[重新挑战] / [返回关卡列表]
```

#### 判定汇总

| 场景 | 判定 | 理由 |
|------|:----:|------|
| 玩家击杀最后一名敌人，敌人死亡效果反杀 | 玩家胜 | 死亡即停，敌人已死效果不触发 |
| 敌人自爆，双方同时全灭 | 玩家负 | 自爆伤害已发出，按正常流程结算 |
| 双方 DOT 在同一结算点同时跳死最后一人 | 平局 | 同一结算点，无时序先后 |

## 三、战斗结束流程

### 3.1 胜利流程

```
判定胜利 → 立即停止行动（中断当前正在进行的动作，不再执行后面的单位）
    ↓
播放胜利表现（字符动画/结算文本）
    ↓
结算奖励计算（经验/金币/掉落）
    ↓
展示结算画面（星级评价 + 奖励清单）
    ↓
玩家选择：[继续]（下一关）/ [返回]（关卡列表/主界面）
```

**"立即停止行动"的含义：**
- 如果胜利发生在行动阶段中途（惊鸿击杀最后一个敌人后触发追击，但已无目标），后面的单位不再执行行动
- 当前轮剩余的出手序列全部取消
- 不执行轮结束阶段（因为战斗已结束）
- DOT/HOT 在本轮不再结算（战斗已在 DOT 触发前或后结束）

### 3.2 失败流程

```
判定失败 → 立即停止行动
    ↓
播放失败表现（"全军覆没"字符提示）
    ↓
弹出选择：[重试] / [撤退]
    ↓
重试 → 回到战前准备（重新确认队伍）
撤退 → 返回关卡列表/主界面
```

**重试和撤退的消耗：**

| 操作 | 体力消耗 | 说明 |
|------|---------|------|
| 重试 | **再次扣除 10 点体力** | 重试 = 重新挑战，消耗资源和进入战斗一样 |
| 撤退 | **体力不退** | 进入战斗时已扣除体力，战败/撤退不返还 |
| 退出小程序 | **体力不退** | 战斗途中退出 = 判负，下次进游戏回到关卡选择 |

设计理由：
- 重试需要再次扣体力，避免玩家无限制 SL（Save/Load）磨死高难关卡
- 撤退不退体力，防止"打不过就撤退避免损失"的钻空子行为
- 单机游戏不需要买体力，但体力是节奏控制工具，不能变成无限

**防误触措施：**
- 重试按钮点击后弹出确认弹窗，确认按钮附带 3 秒倒计时，倒计时结束后才可点击
- 倒计时期间按钮显示"确认重试（3s）"
- 防止玩家想点[撤退]却误触[重试]导致体力浪费

### 3.3 撤退/放弃战斗

**玩家主动撤退**发生在战斗中（非阵亡状态）：

```
玩家在行动阶段点击 [撤退] → 确认弹窗"确定要撤退吗？"
    ↓
确认后 → 战斗结束 → 按失败处理（体力不退）
    ↓
返回关卡列表
```

**限制条件：**
- 只有玩家方有存活单位时才能撤退（全灭直接判负，不需要撤退选项）
- 撤退没有额外惩罚（除了体力消耗）
- 第一回合不能撤退（防止误触）

### 3.4 资源消耗时机

**体力扣除时机：进入战斗时立即扣除，而不是战斗结束时。**

流程：
```
战前准备 → 确认开始战斗 → 扣除体力 × 1（如 10 点）→ 进入战斗画面
    ↓
战斗结束 → 结算（不涉及体力扣除/返还）
```

这个设计意味着：
- 战斗中闪退/断线/关闭小程序：体力已扣，下次进入不返还
- 闪退回游戏后回到关卡选择（战斗状态不保存，详见第六章）
- 体力在进入战前准备时还没扣，玩家可以随意调整队伍

### 3.5 战斗中的强制退出情况

| 场景 | 处理 |
|------|------|
| 玩家关闭小程序 | 判负，体力不退，无结算 |
| 网络异常（虽然单机，但小游戏启动可能需要网络） | 判负，体力不退，回到关卡选择 |
| 异常崩溃 | 同上（单机游戏存档点在前一个界面，体力已扣） |

## 四、战斗结算

### 4.1 结算界面展示内容

```
┌─────────────────────────────────────────┐
│             战  斗  胜  利               │
│                                         │
│  ── 战  斗  数  据 ──                    │
│  用时: 12 轮                             │
│  总伤害: 18920                          │
│  最高伤害: 惊鸿·苏玄 11060              │
│  击杀: 惊鸿·苏玄 × 3 | 方士·青崖 × 1   │
│  总治疗: 3560                           │
│  阵亡: 无                               │
│                                         │
│  ── 奖  励  一  览 ──                    │
│  经验 × 560    游侠等级 +120            │
│  金币 × 300                             │
│  掉落: 技能残卷·方 × 2                  │
│                                         │
│  ★ ★ ☆  评价：2 / 3 星                 │
│                                         │
│  [继续下一关]        [返回关卡列表]      │
└─────────────────────────────────────────┘
```

**设计原则：对单机 IAA（广告变现）游戏来说，结算界面做到"有信息量但不冗余"。**

### 4.2 展示数据的意义

| 数据项 | 展示理由 | 不展示理由 |
|-------|---------|-----------|
| 用时（轮数） | 让玩家对战斗效率有感知，"这次打得快/慢" | — |
| 总伤害 | 成就感，"我打了这么多伤害" | — |
| 最高伤害 | 鼓励 DPS 角色，制造爽感 | 如果每场都显示相同角色，可能看腻 |
| 总治疗 | 让奶妈玩家也有存在感 | — |
| 阵亡统计 | 影响星级评价，需要明确告知 | — |
| 个伤明细 | — | 3 角色不需要逐人展示，信息量过大 |
| 受伤害 | — | 对单机 IAA 游戏过度，玩家不需要分析受伤害 |
| 治疗过量 | — | 只对硬核玩家有意义 |

**结论：展示总伤害、最高伤害、总治疗、击杀数、阵亡统计就够了。** 这是"爽感导向"的结算，不是"数据分析导向"。

**MVP（最高伤害 / 最高治疗）计算规则：**
- 伤害值 = 对敌方单位造成的实际扣血量（排除护盾吸收部分）
- 如果两个角色最高伤害并列，按队伍位置 1 > 2 > 3 的顺序选择 MVP
- 治疗角色单独显示"最高治疗"（与最高伤害并列展示），让奶妈也有成就感

**有效伤害定义：**
- 有效伤害 = 对敌方单位 HP 的实际减少量
- 护盾吸收部分不计入有效伤害（打在盾上不算"伤害"）
- 过量伤害（击杀后溢出的伤害）计入总伤害输出显示，但不计入有效伤害判定
- MVP 统计使用总伤害输出（含过量），界面显示的也是总输出量
- 简单说：MVP 看"打出去的总伤害"（含过量、不含护盾吸收），和敌方实际扣血量是两套口径

### 4.3 失败界面展示

```
┌─────────────────────────────────────────┐
│             全  军  覆  没               │
│                                         │
│  第 8 轮  阵亡                          │
│                                         │
│  太强了...调整阵容再来一次？             │
│                                         │
│  ⚠ 体力已消耗，重试需再次消耗           │
│                                         │
│  [重  试]          [撤  退]             │
└─────────────────────────────────────────┘
```

设计要点：
- 不展示伤害数据（输了看数据没意义）
- 告知体力已消耗（让玩家知情决定是否重试）
- 给一句鼓励文案（不是"你太菜了"）

### 4.4 角色存活状态对经验的影响

**规则：所有参战角色（包括阵亡角色）都获得等量经验。**

理由：
- 如果阵亡角色不获得经验，玩家在高难本中会避免上阵低等级角色（怕他们死、死了白打）
- 这会让"带新人练级"变得不可能——低等级角色更脆，更容易死，死了没经验 → 永远升不了
- 人人平等获得经验，鼓励玩家尝试不同阵容

**例外：如果角色未上场（替补位），不获得经验。** 当前版本无替补位，预留规则。

**失败时的经验获取：**
- 战斗失败时，参战角色获得胜利经验的 30%（向下取整）
- 体力已消耗，但给部分经验作为补偿，避免"打不过 = 没资源提升 = 更打不过"的恶性循环
- 平局时同样获得胜利经验的 30%

## 五、战利品/掉落

### 5.1 基础奖励（每场必得）

| 奖励类型 | 获取条件 | 说明 |
|---------|---------|------|
| 经验值 | 战斗胜利 | 全部参战角色获得等量经验 |
| 游戏币（金币） | 战斗胜利 | 基础数量 × 关卡倍率 |
| 游侠经验（账号经验） | 战斗胜利 | 提升游侠等级 |

基础奖励的数量取决于关卡设计，此处只定义"有"和"什么时候有"。

### 5.2 额外掉落（概率获得）

| 掉落类型 | 来源 | 说明 |
|---------|------|------|
| 技能残卷·玄 | 掉落 | 玄甲技能升级材料 |
| 技能残卷·方 | 掉落 | 方士技能升级材料 |
| 技能残卷·惊 | 掉落 | 惊鸿技能升级材料 |
| 技能残卷·素 | 掉落 | 素问技能升级材料 |
| 装备材料 | 掉落 | 装备强化/制作材料 |
| 招募券 | 掉落（低概率） | 稀有掉落 |

**掉落规则：**
- 掉落在战斗胜利时一并结算
- 关卡配置固定掉落池（每个关卡配置 "必定掉落" + "概率掉落"）
- 多个掉落独立概率判定（互不影响）
- 同一关卡多次挑战，每次掉落独立计算（无保底，除非关卡配置有保底）

### 5.3 首通奖励 vs 重复刷取

| 奖励类型 | 首次通关 | 重复挑战（扫荡） |
|---------|:--------:|:--------------:|
| 基础经验 | 正常获得 | 正常获得 |
| 基础金币 | 正常获得 | 正常获得 |
| 概率掉落 | 正常判定 | 正常判定 |
| 首通额外奖励 | ★ 玉璧/招募券（一次性） | 不获得 |

**首通额外奖励应包括（具体数值由数值策划定）：**
- 玉璧 × 20（每个关卡首次通关固定奖励）
- 招募券 × 1（每章首个关卡首通奖励）
- 装备/材料（特定关卡首通奖励）

### 5.4 扫荡机制

**扫荡条件：** 关卡已获得 ★ 评价（至少打通过一次）。
**扫荡消耗：** 体力 10 点/次（和正常挑战一样）。
**扫荡结算：** 按历史最高星级结算奖励（不用再打一遍战斗）。

扫荡的流程已在主界面文档 4.2 定义，此处确认机制：

```
关卡列表 → 点击已通关关卡 → 弹出扫荡确认
    ↓
确认扣除体力 → 获得基础奖励 + 按历史最高星级 × 奖励倍率
    ↓
展示扫荡结果（简化版结算）
```

**扫荡时各星级对应的奖励倍率：**

| 星级 | 经验倍率 | 金币倍率 | 掉落概率 |
|:---:|:--------:|:--------:|:--------:|
| ★ | ×0.8 | ×0.8 | 基础概率 |
| ★★ | ×1.0 | ×1.0 | ×1.0 |
| ★★★ | ×1.2 | ×1.2 | ×1.2 |

理由：鼓励玩家追求满星，提高资源获取效率。不满星也可以扫荡，只是效率低一些。

### 5.5 掉落不需要在这里做的具体数值

此文档定义机制和规则。具体数值（如"技能残卷·方 掉率 25%"、各关卡金币数量、经验曲线）由数值策划在后续阶段输出。

**需要数值策划决定的项：**

| 项 | 说明 |
|----|------|
| 每关基础经验值 | 与关卡等级挂钩 |
| 每关基础金币 | 与关卡类型挂钩 |
| 各掉落物概率 | 技能残卷、装备材料、招募券 |
| 首通奖励具体数值 | 玉璧、招募券数量 |
| 星级奖励倍率 | 上述 ×0.8/×1.0/×1.2 是否需要调整 |
| Boss 狂暴数值 | +50% 全属性是否合理 |

## 六、战斗状态保存

### 6.1 单机游戏存盘点

**规则：战斗中退出，不保存战斗状态。退出即判负（体力不退）。**

理由：
- 保存战斗状态需要序列化整个 BattleState（出手序列、仇恨矩阵、Buff 状态、当前轮次等），复杂度高
- 微信小游戏的本地存储容量有限（单个 key 上限 1MB），且战斗状态是临时数据
- 一场战斗通常不超过 5 分钟，退出后重打代价不大
- 避免玩家通过"打不过就关掉重进"来 SL 磨死 Boss

### 6.2 如果将来要保存战斗状态

**需要序列化的数据：**

```javascript
class BattleSaveData {
  // 战斗标识
  battleId: string;          // 唯一战斗 ID
  stageId: string;           // 关卡 ID
  roundNumber: number;       // 当前轮数
  currentPhase: string;      // 当前阶段
  currentTurnIndex: number;  // 当前行动索引
  
  // 玩家单位状态
  playerUnits: [{
    unitId: string,          // 角色 ID
    currentHp: number,
    currentShield: number,
    buffs: [EffectInstance], // 活跃 Buff/Debuff
    skillCooldowns: {skillId: remainingCooldown},
    hateTable: {enemyId: hateValue}
  }];
  
  // 敌方单位状态
  enemyUnits: [{
    unitId: string,
    currentHp: number,
    currentShield: number,
    buffs: [EffectInstance],
    skillCooldowns: {skillId: remainingCooldown},
    hateTable: {playerId: hateValue},
    taunted: boolean,
    tauntRemaining: number
  }];
  
  // 当前轮出手序列
  turnOrder: string[];       // unitId 顺序列表
  
  // 其他
  battleLog: string[];       // 最近 N 条战斗日志（用于恢复 UI）
}
```

**当前阶段不保存。** 要存的话需要关注存储大小——3v6 战斗完整序列化预估在 5~10KB，压力不大。但考虑到实现复杂度和产品阶段，建议 v1.0 不做。

## 七、边界情况

### 7.1 DOT 杀死最后一个敌人

**规则：DOT 在轮开始阶段杀死最后一名敌人 → 战斗胜利，不需要等 DOT 跳完。**

处理流程（与 2.1 节轮开始阶段顺序一致）：
```
轮开始阶段：
  1. 胜负判定（兜底） → 无变化
  2. DOT/HOT 结算 → 最后一名敌人 HP ≤ 0
  3. 击杀事件（通灵/追猎检查）
  4. 胜负判定 → 检测到敌方全灭 → 战斗胜利
  （跳过后续的重置标记和领域计时）
```

被 DOT 杀死的敌人，击杀归属：
- DOT 的击杀归属 = 施加该 DOT 的单位
- 如果是方士施加的 DOT 杀死敌人 → 触发通灵（冷却 -1）
- 如果是装备 DOT 杀死敌人 → 击杀归属"装备" → 通灵不触发（需要方士本人击败）

### 7.2 追猎/通灵在胜利前的最后一次击杀

**追猎场景：**
```
惊鸿行动 → 击杀最后一名敌人（非最后一名） → 触发追猎
  → 追猎对随机另一个敌人造成伤害
  → 如果追猎恰好打死最后一个敌人 → 战斗胜利
  → 追猎的击杀归属是惊鸿，通灵不触发（通灵需要方士击杀）
```

**通灵场景：**
```
方士行动 → 烈焰符 AOE → 同时杀死多个敌人（包括最后一个）
  → 每个击杀独立触发通灵（冷却 -1）
  → 最后一个敌人死亡 → 战斗胜利
  → 如果击杀事件已经触发（通灵 CD 缩减已完成），胜利判定不影响已执行的效果
  → 胜利判定之后不再触发新的击杀事件（战斗已结束）
```

**关键规则：通灵的冷却缩减在战斗胜利判定之前生效。**
意思是：如果方士击杀最后一个敌人，通灵触发冷却缩减——这个缩减在战斗结算前已经生效了，下一场战斗（如果连续战斗）技能冷却状态正确。

### 7.3 敌方最后一人自爆/自损杀死己方全体

已在 2.4 节覆盖。总结：玩家击杀最后一人 → 玩家胜（死亡即停，敌人死亡效果不触发）；自爆同归于尽 → 玩家负；同一结算点 DOT 对 DOT → 平局。

补充细节：
- 自爆/自损效果在伤害公式层跟在技能描述中标注 `[自爆]`、`[反伤]` 等标记
- 如果自爆伤害先杀死自爆者自己，再杀死玩家 → 自爆者死亡后其效果是否继续执行？
  - **规则：自爆效果在自爆者死亡后仍然执行**（自爆是瞬间效果，不是持续的）
  - 例如："自己受到 9999 伤害，对所有敌人造成 ATK×300% 伤害"——自爆者先死，但伤害已经发出

### 7.4 Boss 战特殊规则

**多阶段转场不算战斗结束：**

- 某些 Boss 有多阶段（P1 → P2 转场），转场时 Boss HP 可能回满、形态变化、小怪刷新
- **转场不触发胜负判定**——Boss 的"HP ≤ 0"触发的是转场事件，不是战斗胜利
- 转场事件在伤害流程中作为一个特殊 hook：
  ```
  伤害结算 → Boss HP ≤ 0 → 检查是否为转场点
    → 是 → 执行转场（恢复 HP、清除 debuff、召唤小怪等）
         → 战斗继续（不触发胜利判定）
    → 否 → 正常击杀判定
  ```
- 最终形态被击杀后才算战斗胜利
- 转场期间不清除仇恨表（Boss 还记得之前谁打它最狠），但可以清除嘲讽
- 转场后 Buff/Debuff 的处理：
  - 玩家方身上的 buff/debuff 保留
  - Boss 身上的 debuff 是否清除取决于转场配置（"清除所有 debuff"是一种常见转场效果）

**当前阶段：Boss 多阶段转场仅在关卡设计中定义具体配置，本文件只定义框架规则。**

### 7.5 关卡中的连续战斗

**当前设计：一场战斗结束后回到关卡列表/结算界面，不存在"打完一场自动进入下一场"。**

如果需要连续战斗（如 Boss Rush、爬塔），规则如下（预留）：
- 连续战斗的"战斗结束"仅指当前这场战斗结束
- 结束后显示连续战斗的过渡界面（"第 3/10 层"）
- 角色状态**不完全恢复**：HP 和 Buff 保留，技能冷却保留，阵亡角色不回场
- 只有到达连续战斗终点（最终 Boss 击败）才算真正的"战斗胜利"
- 中途撤退：按当前进度结算奖励

**当前阶段：v1.0 不实现连续战斗模式，此处仅作预留接口，避免后续扩展时设计冲突。**

**关于与 6.1 节"不保存战斗状态"的一致性说明：**
- 6.1 节针对的是"战斗中退出"场景——玩家中途关闭小程序，战斗状态不保存
- 连续战斗模式下，战斗状态在内存中跨场次保留，仅在完整连续战斗结束时销毁
- 两者不矛盾：一个防中途退出丢进度，一个防连续战斗间状态丢失
- 连续战斗模式下玩家中途退出 → 按当前进度结算损失（不追溯）

### 7.6 敌方召唤物与胜利条件

如果敌方有召唤技能（当前未设计），召唤物也算敌方单位：
- 胜利条件仍然是"全部敌方单位 HP ≤ 0"，包括召唤物
- 但是等到召唤物自然消失后，可以绕过战斗判断？
  - **规则：召唤物的存在本身不影响胜利判定。** 必须全部消灭。
- 召唤物的击杀归属：造成击杀的单位（而不是召唤者的）

### 7.7 胜利时残留的 DOT/HOT

**场景：DOT 在轮开始阶段杀死最后一个敌人，但同一轮还有其他 DOT/HOT 未结算。**

规则：
- 战斗胜利后，所有未结算的 DOT/HOT 不再触发
- 所有活跃的 buff/debuff 在战斗结束时统一清除
- 不需要逐层移除（直接清空战斗状态）

### 7.8 仇恨表在战斗结束时的清除

战斗结束时所有仇恨表清空。不存在"这场的仇恨带到下一场"的情况。

## 八、与游戏流程的衔接

### 8.1 战斗进入

参照主界面文档中定义的流程：

```
主页 → 闯关 → 章节列表 → 关卡列表 → 战前准备
  → 确认队伍 → [开始战斗]
    → 扣除体力（1 次）
      → 初始化战斗 → 进入战斗画面
```

**战斗初始化时需要的参数：**

| 参数 | 来源 | 说明 |
|------|------|------|
| 关卡配置 | 关卡表（stage_config） | 敌方阵容、轮数上限、星级条件、掉落池 |
| 玩家队伍 | 当前编队 | 3 个出战角色的完整配置（等级、技能、装备） |
| 随机种子 | 当前时间戳 | 用于重现战斗（如果需要） |

**关于随机数生成器（RNG）：**
- 从开发初期开始，所有战斗中的随机判定（暴击、伤害波动、追猎随机目标、掉落概率等）必须使用基于种子的 PRNG（伪随机数生成器），而非 `Math.random()`
- 即使当前不实现战斗录像，种子 PRNG 为将来留了后路，实现成本极低
- 推荐实现：
  ```javascript
  // 战斗初始化时
  const battleSeed = Date.now();
  const rng = new SeededRandom(battleSeed);
  // 所有随机判定使用 rng.next() 而非 Math.random()
  ```
- 一个 SeededRandom 类约 30-50 行代码，可使用简单的 Mulberry32 或 xoshiro128 算法

## 九、星级评价（正式化）

### 9.1 条件定义

来自主界面文档 4.4，此处正式化为战斗接口：

| 条件 | 星级 | 判定时机 | 说明 |
|------|:----:|---------|------|
| 战斗胜利 | ★（基础） | 战斗结束时 | 只要赢了就有 |
| 无人阵亡 | ★ | 战斗结束时 | 全战斗过程中没有一个己方单位 HP 归零 |
| 10 轮内结束 | ★ | 战斗结束时 | 从第 1 轮开始到战斗结束 ≤ 10 轮 |

满星 3 星。扫荡时按历史最高星级结算奖励。

### 9.2 "无人阵亡"的判定要点

- 只要有一个己方单位在战斗中 HP 降为 0，就判定为"有阵亡"
- 包括复活后仍算阵亡过（当前无复活技能，预留规则）
- 召唤物/替身的阵亡不算角色阵亡
- 判定在战斗结束时检查"是否有阵亡记录"，不是"当前是否全员存活"

**阵亡记录数据结构：**

```javascript
class CombatStarTracker {
  hasCasualty: boolean;                 // 全局标记：是否有己方阵亡
  unitCasualties: Map<string, boolean>; // 按单位追踪：unitId → 是否阵亡过
}

// 使用方式：
// 1. 战斗初始化时：hasCasualty = false，unitCasualties 清空
// 2. 任何己方单位 HP 降至 0 时：hasCasualty = true，unitCasualties.set(unitId, true)
// 3. 战斗结束时检查：星级判定时读取 hasCasualty
// 4. 当前无复活技能，如果将来加入复活：复活后 hasCasualty 保持 true（已阵亡记录不清除）
```

### 9.3 "10 轮内结束"的判定要点

- 从第 1 轮开始计数
- 进入第 11 轮时自动判定为不满星（< 10 轮 = 在第 10 轮结束之前完成）
- 如果战斗在第 10 轮的行动阶段结束 → 算 ≤ 10 轮，得星
- 如果战斗在第 11 轮的轮开始阶段结束（DOT 杀死）→ 算 11 轮，不得星
- 教学关无上限但不代表自动满星——教学关如果满足条件也有 3 星

### 9.4 星级与掉落的关系

具体倍率见 5.4 节扫荡机制。手动战斗的星级不影响当场掉落（打的时候掉落已按关卡基础概率判定），但扫荡时受星级影响。

## 十、战斗日志（调试与回放）

### 10.1 日志格式建议

```javascript
// 战斗结束后保存的简要日志
class BattleSummary {
  stageId: string;
  result: 'victory' | 'defeat';
  totalRounds: number;
  starRating: number;            // 0-3
  casualties: string[];          // 阵亡角色 ID 列表
  totalDamageDealt: number;
  totalDamageTaken: number;
  totalHealing: number;
  mvpUnitId: string;             // 最高伤害单位
  drops: [{itemId: string, count: number}];
  detailedLog: BattleLogEntry[]; // 详细行动记录（可收起）
}
```

### 10.2 详细日志格式

```javascript
class BattleLogEntry {
  round: number;
  phase: 'ROUND_START' | 'ACTION' | 'ROUND_END';
  unitId: string;           // 行动单位
  action: string;           // "skill:破甲箭" / "normal" / "defend"
  targetId: string;         // 目标单位
  damage?: number;          // 伤害值
  healing?: number;         // 治疗值
  isCrit?: boolean;         // 是否暴击
  isKill?: boolean;         // 是否击杀
  hateChange?: {enemyId: string, change: number};  // 仇恨变化
  buffApplied?: string[];   // 施加的 buff
  buffRemoved?: string[];   // 移除的 buff
}
```

详细日志仅在调试模式/回顾时查看，不需要在结算界面展示。

**日志存储策略：**
- 全局最多保留最近 20 场战斗的日志（20 × 50 = 1000 条条目上限）
- 超过上限时淘汰最旧的战斗日志（FIFO 策略）
- 存储键名格式：`battle_log_${timestamp}`，按时间戳命名方便清理
- 调试模式下可永久保留，正式发布版严格执行存储上限
- 日志仅保存到本地存储（wx.setStorageSync），不涉及网络传输
