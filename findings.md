# 发现与决策

## 需求
- 微信小程序回合制对战+角色养成单机游戏
- 中世纪奇幻+中国古风审美
- 极简字符风格美术
- 小队回合制（我方3人 vs 敌方1-6+）
- 四大职业：玄甲（坦）、方士（法）、惊鸿（弓）、素问（奶）
- 仇恨系统替代站位
- 有具体角色名（中国古风），固定星级（四/五星），招募获得

## 团队架构决策
- 使用 Claude Code Agent 系统创建不同游戏设计角色
- 每个 agent 有独立的系统提示词和专业领域
- 人员精简，核心 agent 先行

## 技术决策
| 决策 | 理由 |
|------|------|
| 使用 agent-develop 技能创建自定义 agent | Claude Code 原生支持 agent 架构 |
| 先建策划类 agent，后建技术类 | 项目处于设计阶段，开发尚早 |
| 微信小程序原生框架（非 uni-app/Taro） | 前端零基础：少一层抽象少一层坑；字符风格游戏不需要跨平台；中文资料最多；包体积最小 |

## 遇到的问题
| 问题 | 解决方案 |
|------|---------|
| 多个 agent 如何协作？ | 由主 Claude 担任主策划/统筹角色，协调各 agent 产出 |
| agent 间上下文不共享 | 通过 planning-with-files 文件系统和主 Claude 传话 |

## 项目定位决策
| 决策 | 理由 |
|------|------|
| 定位为练习项目 | 不以赚钱为首要目的，降低心理包袱 |
| 个人开发者，不走企业路线 | 不做内购，不做竞技/联机，避开版号 |
| 变现方式：广告（IAA） | 唯一对个人开放的变现渠道，有最好没有也无所谓 |
| 先做核心玩法，再考虑发布 | 降低前期投入风险，做出来满意了再走审核流程 |

## 待办设计提醒
- 速度属性是纯排序属性，数值宽度无意义。后续装备/天赋投放速度词条时要注意，避免"堆了没生效"或"没堆被插队"的体验
- 行动阶段内改变速度的效果应**下一轮才生效**，实现时需统一遵守
- **职业初始属性表需要平衡**：当前数值下，方士 ATK 偏高导致其 AOE 倍率技能的伤害超过惊鸿单体爆发，惊鸿的职业定位被挤压。建议降方士 ATK 或提惊鸿倍率
- **治疗量基准待定**：150% 治疗倍率下，素问一次治疗只回坦 8.75% HP，偏弱。可能需要基础倍率提到 250-300% 或靠装备补
- **DEF_CONST 模型**：防御的 EHP 收益是线性而非递减的（每点 DEF 等量提升有效生命），做装备投放时不要误导玩家说"堆防收益递减"
- 等级补正配合属性成长，10 级差实际伤害差距约 2 倍（并非文档说的"温和"），跨级挑战需靠策略弥补
