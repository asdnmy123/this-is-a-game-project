# 发现与决策

## 需求
- 微信小程序回合制对战+角色养成单机游戏
- 中式古风奇幻（大晟王朝/江湖+方术）
- 极简字符风格美术
- 小队回合制（我方3人 vs 敌方1-6+）
- 四大职业：玄甲（坦）、方士（法）、惊鸿（弓）、素问（奶）
- 仇恨系统替代站位
- 有具体角色名（中国古风），固定星级（四/五星），招募获得

## 团队架构决策
- 使用 Claude Code Agent 系统创建不同游戏设计角色
- 每个 agent 有独立的系统提示词和专业领域
- 人员精简，核心 agent 先行

## 命名与世界观决策
| 决策 | 理由 |
|------|------|
| 游戏名《八方游侠》| 配套 UI"游侠阁"，与招募/编队玩法衔接 |
| 世界观走纯中式路线（大晟王朝/江湖+方术）| 风格统一，避免"中不中西不西"的撕裂感 |
| 世界观"中世纪奇幻"描述正式废弃 | 已过时，统一为"中式古风奇幻" |

## 技术决策
| 决策 | 理由 |
|------|------|
| 使用 agent-develop 技能创建自定义 agent | Claude Code 原生支持 agent 架构 |
| 先建策划类 agent，后建技术类 | 项目处于设计阶段，开发尚早 |
| 微信小程序原生框架（非 uni-app/Taro）→ Cocos Creator | 最初选择微信小程序原生框架，后因需要先上 iOS/Android 再扩展微信小游戏，改为 Cocos Creator（2D 游戏引擎，TypeScript，支持三端导出） |
| 字符风格游戏不需要跨平台 → 需要跨平台 | 最初定位仅限微信小游戏，改为 iOS/Android 为主、后续微信小游戏，跨平台成为刚需 |

## 遇到的问题
| 问题 | 解决方案 |
|------|---------|
| 多个 agent 如何协作？ | 由主 Claude 担任主策划/统筹角色，协调各 agent 产出 |
| agent 间上下文不共享 | 通过 planning-with-files 文件系统和主 Claude 传话 |

## 项目定位决策
| 决策 | 理由 |
|------|------|
| 定位为练习项目 | 不以赚钱为首要目的，降低心理包袱 |
| 个人开发者，不走企业路线 | 不做内购，不做竞技/联机，避开版号 |
| 变现方式：广告（IAA） | 唯一对个人开放的变现渠道，有最好没有也无所谓 |
| 先做核心玩法，再考虑发布 | 降低前期投入风险，做出来满意了再走审核流程 |

## 设计文档管理原则

| 决策 | 理由 |
|------|------|
| "定稿"非冻结，可修改 | 定稿意思是"当前可用、下游可依赖"，不是封死不改。游戏设计牵一发动全身，后面发现不合理必须能回头改 |

## 平台决策

| 决策 | 理由 |
|------|------|
| 技术栈改为 Cocos Creator | 需要先上 iOS/Android APP，后续导出微信小游戏。Cocos 是唯一同时满足：2D 游戏轻量 + 三端导出 + 微信小游戏一等公民支持的引擎 |
| 放弃微信小程序原生框架 | 原生框架只能跑在微信里，无法上架 App Store/安卓商店，与"先做 APP"的目标冲突 |
| Cocos Creator 优于 Unity | 包体小（对小游戏 4MB 限制友好）、2D 渲染管线轻量、微信小游戏一键导出。Unity 虽然开发者生态更大，但在这个项目上属于过度杀伤 |
| 不排除技术上用 TypeScript 而非 Cocos 原生语言 | 最终实现阶段再确定具体方案 |
| 小改直接改，顶部留 changelog | 改数字、调表述这类不影响核心逻辑的改动，直接编辑文档，加条修改记录即可 |
| 大改走重新评审流程 | 推翻核心逻辑需要：抛问题 → agent 出修改方案 → 评审确认 → 更新状态标记 |
| 改完必须检查下游依赖 | 改了某份设计，所有依赖它的未完成设计都要同步更新。task_plan.md 的依赖关系做这个用 |

## 计划管理原则

## 招募系统设计决策

| 决策 | 理由 |
|------|------|
| 3★ 不设计角色，只做材料填充 | 上版本只有 4★/5★ 角色，3★ 放养成材料避免卡池太空 |
| 5★ 综合概率约 2.08%（90 抽硬保底） | 单机友好，平均 3 周一个 5★，不逼氪 |
| 每日免费 1 抽 + 剧情送 3 个 4★ | 零氪玩家也能组完整队伍推图 |
| 命座 + 升星双轨重复处理 | 4★ 可升 5★ 继续使用，真爱党不亏 |
| 命座效果全为属性 + 被动强化 | 机制不变只加数值，避免"没有某命座就不能玩" |
| 招募资源以"周"为周期估算 | 符合轻度玩家上线节奏，不追求日活 |

## 主线剧情修复决策

| 决策 | 理由 |
|------|------|
| 第5章结尾大长老将玉璧碎片交给玩家 → 第6章暗线黄天教趁乱盗走 | 填补"第7章黄天教集齐5块"的逻辑缺口：玩家在第5章拿到碎片，第6章玩家主力不在阁中时被盗，逻辑通顺 |
| 预言定义为"灵主重临，净化世间"（黄天教宣传口号） | 不再留空头支票，同时与第7章"灵主非神祇而是人造守护神"的真相呼应——"净化"实为毁灭 |
| 孤狼战后重伤逃离而非战死 | 为后续剧情（角色故事/新章节）预留接口，避免一次性消耗角色 |
| 灵均出自《离骚》"字余曰灵均" | 体现教主文化素养和"替天行道"的自我定位，符合中式古风基调 |

## 待办设计提醒
- 速度属性是纯排序属性，数值宽度无意义。后续装备/天赋投放速度词条时要注意，避免"堆了没生效"或"没堆被插队"的体验
- 行动阶段内改变速度的效果应**下一轮才生效**，实现时需统一遵守
- **职业初始属性表需要平衡**：当前数值下，方士 ATK 偏高导致其 AOE 倍率技能的伤害超过惊鸿单体爆发，惊鸿的职业定位被挤压。建议降方士 ATK 或提惊鸿倍率
  → **已解决**：四大职业设计方案中，方士 ATK 从 560 降至 500，修正后惊鸿单体期望伤害反超方士
- **治疗量基准已定**：已输出 `docs/combat/2026-05-28-treatment-baseline.md`（draft）。建议普攻 80%→100%，回春术 200%→320%，益气汤 120%→180%，悬壶 30%→40%，配合装备治疗加成曲线。技能系统设计时需同步更新 class-design.md 中的倍率数值
- **玉璧经济可持续性**：招募方案中玉璧收支为净亏损（周收 550 支出 1000），需在后续经济系统设计中闭环。见 docs/system/2026-05-28-recruitment-design.md 5.4 节的 warning
- **DEF_CONST 模型**：防御的 EHP 收益是线性而非递减的（每点 DEF 等量提升有效生命），做装备投放时不要误导玩家说"堆防收益递减"
- **等级补正配合属性成长，10 级差实际伤害差距约 2 倍（并非文档说的"温和"），跨级挑战需靠策略弥补

## 仇恨系统设计决策

| 决策 | 理由 |
|------|------|
| 玄甲仇恨倍率 ×3（初值，可调） | 玄甲伤害低（DEF转换），×3确保基础仇恨但不锁死，DPS爆发期可OT |
| 嘲讽仇恨硬加成 +500 | 让嘲讽结束后玄甲仇恨仍领先，但空窗期DPS持续输出可超车 |
| 衰减率 80%/轮（比例衰减） | 比例衰减对大/小数值自适应，3轮剩50%，防止仇恨永久固化 |
| 治疗/护盾产生"全体仇恨" | 实现简单（不需要追踪每个敌人是否看到），世界观也说得通 |
| Boss 部分/完全免疫嘲讽 | 让Boss战有操作感，坦克不能无脑嘲讽挂机，需要堆输出维持仇恨 |
| 仇恨逃逸上限 3.0x（有玄甲且目标仇恨>0时生效） | 从2.5x上调以给OT留空间；无玄甲/玄甲未碰过的目标不设限，避免死锁 |
| AOE仇恨是方士的"软肋" | AOE打多个敌人=总仇恨高，脆皮方士容易引火烧身，需要坦克配合保护 |

## 技能系统设计决策

| 决策 | 理由 |
|------|------|
| 角色技能等级（SLv 1-10）统一管理 4 技能 | 减少管理负担（12→3），避免"只升破甲箭"的畸形配装，保持技能组合完整性 |
| 技能升级不影响冷却 | 冷却固定确保队伍节奏稳定，升级只提升数值 |
| 技能等级上限受角色等级限制 | floor(角色等级/5)，Lv50 解锁 SLv10，避免跳过养成直接堆技能 |
| 护盾不叠加（新盾替旧盾） | 叠加盾过于强力且管理复杂，替换制简单清晰 |
| 护盾先经伤害公式减伤再吸收 | 护盾 = 减伤后的额外吸收层，与百战之躯的减伤自然叠加 |
| Buff 同类不叠加（取最高） | 防止相同效果叠出超模数值，简化计算 |
| Debuff 同类不叠加（刷新时长） | 与 Buff 规则一致，但刷新时长而非取最高（控制节奏更重要） |
| 自动战斗 AI 按条件优先级决策 | 每个技能有明确的使用条件，类行为树风格，可预测可调试 |
| 追猎每轮最多一次 + 不可连锁 | 防止击杀滚雪球过强；普攻 60% 追击是"添头"不是核心输出 |
| 通灵击杀减 CD（无视上限） | 鼓励方士补刀而非只 AOE；Boss 战无小兵时通灵不生效（需后续考虑补偿） |
| 净化优先解控 > 解生存威胁 > 解属性减益 | 回春术的净化按此优先级选目标身上的 debuff，AI 模式自动执行 |
| 悬壶在素问行动结束时触发 | 如素问被控制/击杀则不触发，增加了控/杀素问的战略价值 |

## Buff/Debuff 系统设计决策

| 决策 | 理由 |
|------|------|
| DOT/HOT 在轮开始阶段触发（而非轮结束） | DOT 在目标行动前扣血，可能跳过其本轮行动；HOT 在目标行动前回血，以更好状态进入行动阶段 |
| DOT 不能暴击 | 持续性伤害的优势是"稳定每轮触发"，与暴击的"不稳定高爆发"是两套体系，不应混用 |
| 效果在施放者死亡后保留（嘲讽除外） | 避免玩家困惑"我刚上的buff怎么没了"；嘲讽与玄甲强绑定，玄甲死嘲讽消 |
| 效果在目标死亡后立即移除 | 死人不应该有 buff，DOT/HOT 在死者身上无意义，防止复活异常 |
| 没有命中率/闪避系统，所有效果必中 | 不引入额外属性维度；控制成功与否取决于免疫而非随机；玩家可清晰预判 |
| 眩晕 > 冰冻 > 沉默（控制强弱排序） | 多控制共存时取"限制更强的"实际生效，各自独立计时 |
| 同 ID 相同控制刷新时长，不同 ID 不同控制各自独立 | 与属性类 buff 的 refresh 规则一致，统一行为 |
| Boss 控制免疫分级（完全免疫/部分免疫减半） | 弹性设计让不同 Boss 有不同控制抗性，给玩家"这个 Boss 吃不吃控"的探索空间 |
| UI 最多显示 6 个 buff 图标，超限隐藏 | 微信小屏幕空间有限；优先级：控制 > DOT/HOT > 属性 > 其他 |
| DOT/HOT 最多 3 层（相同 ID 的） | 防止无限叠层导致伤害/治疗失控；同源不叠层（同 ID 刷新） |
| DOT/HOT 不产生每轮仇恨 | 仇恨在施放技能那一刻已产生，持续效果不应重复产生仇恨 |
| buff+debuff 对冲直接代数和，不分先后 | 简化计算，彼此抵消自然；无"谁优先级高"的问题 |
